# Dialogues —— 关键对话存档

本目录存放真正塑造判断的、值得长期回看的人机对话。

## 收录标准

参见 [`../CHARTER.md`](../CHARTER.md) 中的"收录原则"。至少满足两条：

1. **决策被这次对话改变了**。
2. **会被引用** —— 三个月后做相关决策时仍可能回头查。
3. **包含可被抽象的结构** —— 一次具体讨论里浮现出的、可在其他场景复用的方法。

## 命名约定

`YYYY-MM-DD-<short-slug>.md`

例如：

- `2026-05-21-naming-mithaq.md` —— 关于本仓库命名过程的对话
- `2026-04-30-checkpoint-mechanism-generalization.md` —— 关于把 roboclaws 的 checkpoint 机制抽象成跨 repo 模板的对话

slug 用动词或名词短语，描述这次对话**让什么发生了改变**，而不只是描述聊了什么话题。

## 文件结构建议

每个 dialogue 文件用如下结构（不强制，跑两三份找到自己的节奏后再固化）：

```markdown
# {{短描述：这次对话改变了什么决定}}

## 元数据

- **日期**：YYYY-MM-DD
- **模型**：Claude Opus 4.7 / GPT-5.x / ...
- **触发问题**：{{开始这次对话的最初问题}}
- **影响的 repo**：{{哪些 repo 的判断因此变化}}
- **状态**：{{进行中 / 已落地 / 已废弃}}

## 关键结论

> 3-5 条，让没读原文的人也能拿到核心。

- ...

## 可抽象的结构（如有）

> 这次具体讨论里浮现出的、可在其他场景复用的方法 / 框架 / 决策模式。

## 对话节录

> 不必照抄全部。截取关键的几个回合即可。
> 上下文太多时用 `[...]` 表示省略，并简要写省略段落讲了什么。

### 用户

...

### Claude

...

## 后续行动

- ☐ 落地到 `<repo>` 的 issue #N
- ☐ ...

## 修订

> 后来发现某个判断错了 / 不再成立时，在这里记。不删除原文，保留判断的演化。

- {{YYYY-MM-DD}}: {{...}}
```

## 隐私 / 安全

- **不要存包含未公开 API key、token、secret 的对话**。如果对话里出现这些，先脱敏。
- **不要存包含未公开商业敏感信息的对话**。
- 本目录公开，对话也将公开 —— 当作"将来会发到公众号"的标准对待。

## 为什么不直接用 Claude.ai / ChatGPT 的导出功能

平台导出有几个问题：

1. 格式各异，跨平台无法统一搜索。
2. 平台可能改变历史可见性或删除老对话。
3. 平台导出包含太多无关信息（系统消息、tool call 细节），噪音大。
4. 最关键：**导出是机械动作；记下来需要判断**。`mithaq` 的价值不在于"全部记录"，在于"被选择记录的那部分"。
